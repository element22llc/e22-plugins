# Workflow Spec v0.3 — Revision Design

**Date:** 2026-05-25
**Author:** Alexis Valotaire (via Claude brainstorming session)
**Target spec:** `docs/collaborative-ai-workflow-spec.md` (currently v0.2.1)
**Companion files affected:** `CONSTITUTION.md`, `TECH-STACK.md`, `CONNECTORS.md`, `README.md`, all plugin commands/skills/agents, `marketplace.json`

---

## 1. Why this revision

v0.2.1 was written assuming Vercel + Neon (prototype) and AWS + Terragrunt (production) were the platform substrate. Two practical problems:

1. **Vercel PRO is not free for contributors.** Every contributor needing a PRO account is a non-starter for an org-wide workflow.
2. **Infrastructure is out of scope right now.** The platform layer will be designed and bought separately; the workflow spec should not pre-commit decisions there.

This revision strips infrastructure entirely, simplifies the enforcement model accordingly, and adds three pragmatic features the workflow needs at its current maturity:

- Project-type-aware prototype lane (greenfield vs brownfield)
- Minimum test floor (one smoke test per new endpoint/screen/job)
- Plugin audit against community/Anthropic alternatives

## 3. The v0.3 shape (high-level)

- **Four enforcement layers, not five.** App Guards (runtime boot assertions) drops — it was the infra layer. Remaining: AI Instructions, Repo Contracts, CI Policies, Review Rules. Every rule still maps to ≥1 hard layer; "hard" now means CI or GitHub-enforced.
- **No isolation invariant.** Old invariant #1 and §9.9 (Runtime Guarantees) removed in full. The spec is about *process discipline* now.
- **Single distribution path.** Marketplace install via Claude workspace org plugins, auto-updated on push to `e22-plugins` main. No in-repo snapshot. Accepted cost: cloud agents pull from the marketplace each session.
- **Project-type-aware prototype lane.** Detection writes to `branch.yaml#project_type`; affects scaffolding step only.
- **Min-test floor.** Same rule both lanes, soft/hard split by lane.
- **One plugin slim (security-rails), one new dependency (Anthropic security-guidance).** Other six customs stay.

The one-line summary drops the "Let the infrastructure refuse to lie" clause:

> **Let the PO vibe. Let Claude translate. Let the Dev industrialize.**

## 6. New spec subsections (full text targets)

### §9.9 Project-Type Detection

- **Trigger.** At branch creation, before code generation. Fires from `/vibe` (prototype-lane) or when Claude detects a fresh branch with no `branch.yaml#project_type` set.
- **Detection rules, in order:**
  1. Has a manifest file (`package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`, `mix.exs`, `Gemfile`) **and** non-empty source tree → **brownfield**.
  2. Has a `CLAUDE.md` declaring conventions → **brownfield** (regardless of size).
  3. Empty / only README+LICENSE / freshly `git init`-ed → **greenfield**.
  4. Ambiguous → ask the PO once: *"Is this an existing project I should adapt to, or fresh ground?"*
- **Persistence.** Result written once to `/.workflow/branch.yaml#project_type`. Not re-evaluated unless explicitly reset.
- **Behavioral split (scaffolding only):**

  | Step | Greenfield | Brownfield |
  |---|---|---|
  | Stack selection | Pick from TECH-STACK.md preferences | Read existing manifest, conform |
  | Project layout | Scaffold standard structure | Place new files where existing patterns do |
  | Test runner | Install + configure + write first smoke test | Hook into existing runner; never add parallel |
  | Linter/formatter | Install house defaults | Read existing config; never replace |
  | First commit | `chore: scaffold <stack>` before feature work | First commit is feature work |

- **Lives in:** new agent `plugins/prototype-lane/agents/project-type-detector.md`. Called by the existing `intake-clarifier` agent before routing to scaffolding.

### §9.10 Minimum Test Floor

- **The rule.** Every new endpoint, screen, or background job ships with at least one smoke test exercising its primary path.
- **Detection.** Post-edit, `always-test` scans the diff for new files matching endpoint/screen/job heuristics:
  - Route handler patterns (Express, Next.js, FastAPI, Flask, Rails routes)
  - Component files in `pages/`, `views/`, `app/`
  - Files in `jobs/`, `workers/`, `tasks/`
  For each detected artifact, checks for a co-located or conventional sibling test file (`*.test.*`, `*.spec.*`, `tests/<name>`).
- **Enforcement, lane-scaled:**
  - **Prototype lane:** missing smoke test → soft warning (PostToolUse hook) + Stop-hook reminder at session end. `/package-handoff` refuses to package if no tests run successfully on the branch.
  - **Production lane:** missing smoke test → hook blocks (exit 2). CI fails on coverage delta if untested artifact lands.
- **Greenfield wrinkle.** First test on a greenfield branch counts as "test runner works"; subsequent endpoints/screens trigger the per-artifact rule normally.
- **Lives in:** `plugins/always-test/hooks/check-test-coverage.sh` (detection); `plugins/always-test/hooks/remind-smoke-test.sh` (Stop-hook nudge); `plugins/prototype-lane/commands/package-handoff.md` (refusal check).
