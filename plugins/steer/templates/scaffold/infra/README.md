# Infrastructure

AWS infrastructure-as-code for this product.

This template uses **[OpenTofu](https://opentofu.org/)** as the IaC engine and **[Terragrunt](https://docs.terragrunt.com/)** as the orchestrator and DRY layer. Product teams that want to revisit this choice should write an ADR under `/spec/decisions/` before changing it.

## Toolchain

Tool versions are pinned in [`infra/mise.toml`](./mise.toml). With mise activated, `cd`-ing into `infra/` (or any subfolder) picks up the right `opentofu` and `terragrunt` binaries automatically. The root `mise.toml` stays stack-agnostic so non-infra contributors do not need OpenTofu/Terragrunt on their machines.

```bash
cd infra
mise install        # installs opentofu + terragrunt at the pinned versions
mise lock --platform linux-x64,macos-arm64   # FIRST ADOPTION ONLY: per-platform url+checksum so
                    # CI (linux-x64) can `mise install --locked`. Add other platforms; commit it.
tofu version
terragrunt --version
```

Terragrunt 1.0+ defaults to invoking the `tofu` binary on `PATH`, so no explicit binding is required. The `root.hcl` example below sets `terraform_binary = "tofu"` anyway, for clarity in code review.

## Layout

```text
/infra
├── README.md
├── mise.toml                       # pins opentofu + terragrunt for this folder
├── live/                           # infrastructure-live: deployable units
│   ├── root.hcl                    # remote_state, provider generation,
│   │                               # terraform_binary = "tofu"
│   ├── _envcommon/                 # per-component shared config,
│   │                               # included by leaf units
│   ├── nonprod/
│   │   ├── env.hcl                 # env-scoped variables
│   │   └── <region>/<category>/<component>/terragrunt.hcl
│   └── prod/
│       ├── env.hcl
│       └── <region>/<category>/<component>/terragrunt.hcl
├── modules/                        # OpenTofu modules used by /live
│   └── <component>/
│       ├── main.tf
│       ├── variables.tf
│       └── outputs.tf
└── scripts/
    └── deploy.sh
```

The layout follows the Gruntwork *infrastructure-live* convention: `live/` holds Terragrunt units that describe **what to deploy where**, and `modules/` holds the OpenTofu modules that describe **how to deploy each thing**.

- A **unit** is any directory containing a `terragrunt.hcl` file.
- A **stack** is a collection of units. We start with *implicit* stacks (the folder structure) and only adopt explicit `terragrunt.stack.hcl` files when a clear repeating pattern emerges across environments.

### Scaffolding this layout

This template intentionally ships no `.hcl` or `.tf` files. Each product team creates its own `live/root.hcl`, `live/<env>/env.hcl`, and module files when they are ready to provision real infrastructure. The shape above is the target; the contents are product-specific.

## Conventions

* **Secrets** live in SSM Parameter Store (`SecureString`) by default — cheaper than Secrets Manager and enough for most needs; reach for Secrets Manager only when you need rotation, cross-account sharing, or large/binary values. Never commit credentials, API keys, or `.tfvars` files with secrets.
* **State backend** is S3 + DynamoDB locking, generated by `root.hcl` so every unit gets identical settings. The bucket and lock table are bootstrapped once per environment (manually or via a small `bootstrap/` unit) and then referenced by `root.hcl`.
* **Provider lockfiles** (`.terraform.lock.hcl`) are committed for reproducibility.
* **Infrastructure changes** require dev review like any other PR. The High-risk areas section of `CLAUDE.md` applies — broad infra changes need explicit scoping.
* **Database migrations**, if used, are version-controlled and reviewed explicitly before production release.
* **Modules** live in `/infra/modules` (in-repo) by default. If modules grow beyond what one product can manage, split them into a dedicated `<product>-infrastructure-modules` repo and reference them via `source = "git::..."` from each unit.

## How leaf units wire up

Each `terragrunt.hcl` leaf unit typically:

1. Includes the root config:
   ```hcl
   include "root" {
     path = find_in_parent_folders("root.hcl")
   }
   ```
2. Reads env-scoped values:
   ```hcl
   locals {
     env = read_terragrunt_config(find_in_parent_folders("env.hcl"))
   }
   ```
3. Points at the OpenTofu module:
   ```hcl
   terraform {
     source = "${get_repo_root()}/infra/modules/<component>"
   }
   ```
4. Passes inputs:
   ```hcl
   inputs = {
     env  = local.env.locals.env_name
     name = "..."
     ...
   }
   ```

`root.hcl` typically defines the `remote_state` block (S3 + DynamoDB), a `generate` block for the AWS provider, and `terraform_binary = "tofu"`. Children inherit all of it through `include "root"`.

## Multiple apps, multiple deploy targets

This repo is an internal monorepo: `/apps` may hold more than one deployable
application (e.g. `apps/web`, `apps/api`). Each app is an independent deploy
target with **its own non-prod surface** — its own service/component units under
`live/<env>/`, and it can be deployed without redeploying the others. `/packages`
and `/configs` are never deployed on their own; they ship as part of an app that
depends on them.

Model each app as its own `<component>` (or category) under
`live/<env>/<region>/`, so a change to one app deploys only that app. Shared
infrastructure (VPC, databases, secrets) that several apps consume lives in its
own units and is referenced via remote state.

## Environments

* **non-prod** — shared validation environment. Each app deploys here automatically on merge to `main`. POs validate changes on the relevant app's non-prod surface before production.
* **prod** — production environment. Each app deploys here automatically on merge to the long-lived **`prod` branch**; that merge is gated by a reviewed PR (see Release flow). Apps may be released independently.
* **review apps** — each open feature PR gets an isolated, automatically-provisioned environment so the change can be exercised in a deployed-like context, torn down when the PR merges or closes. The mechanism (per-PR ephemeral stack, shared cluster with PR-scoped namespacing, a preview platform, …) is product-specific — record the choice in an ADR under `/spec/decisions/`.

## How to extend

Single account, single region by default.

* **Multi-region:** add more `<region>/` directories under each env.
* **Multi-account:** wrap `nonprod/` and `prod/` in an `<account>/` parent (e.g., `nonprod-acct/`, `prod-acct/`) and add `account.hcl` at the account scope. Many teams isolate by account instead of by env.
* **Shared org resources:** put account-wide or region-wide resources in `_global/` siblings (e.g., `live/<account>/_global/`, `live/<account>/<region>/_global/`).

## Release flow & promotion

Deployment is **branch-driven** — promotion happens by moving code between
branches, not by manually triggering environments. This gives a real production
approval gate without GitHub Enterprise (whose deployment-environment approvals
aren't available on this plan): the gate is the required PR review on `prod`.

1. **Feature PR opened** → a **review app** is provisioned for it; the PO/dev
   exercise the change there.
2. **PR merged to `main`** → the change **auto-deploys to non-prod**; the review
   app is torn down.
3. **PO validates** the change on non-prod and signals readiness (e.g. `👍 ready
   to ship` on the issue/PR).
4. **Promotion PR `main` → `prod`** is opened and **approved** by a reviewer. This
   approval is the production gate — no direct pushes to `prod` (enforced by
   `/steer:protect`; see the `prod` entry in `policy/branch-protection.yml`).
5. **PR merged to `prod`** → the change **auto-deploys to prod**. Apps may be
   promoted independently.
6. **Deployment result is posted back** to the issue/PR.

**Rollback.** Every prod deploy has a known rollback: revert the `prod` merge (the
next auto-deploy returns prod to the prior state) or redeploy the previous SHA.
Database migrations are **expand/contract** — additive first, so the currently
running version keeps working through the deploy and a rollback never leaves the
schema ahead of the code. Do not deploy risky changes to production on a Friday
unless there is an explicit reason and a named rollback owner.

## Observability

A deployed environment is not done until it is observable. Baseline, wired in
`/infra` and recorded in `ARCHITECTURE.md`:

* **Logs** — application + platform logs shipped to a queryable store (e.g. CloudWatch Logs).
* **Metrics & alarms** — service and host metrics with alarms on the signals that matter (error rate, latency, saturation), routed to where a human sees them.
* **Error tracking** — Sentry (or equivalent) capturing exceptions with context on frontend and backend.
* **Health checks** — a liveness/readiness endpoint per app, wired to the load balancer / orchestrator and to uptime alerting.
* **Alerting** — a defined destination (on-call, channel) for alarms and error spikes; an alarm nobody receives is not monitoring.

## Useful commands

```bash
# Pick an env + region + component (a "unit"):
cd live/nonprod/<region>/<category>/<component>

terragrunt init
terragrunt plan
terragrunt apply

# Run across all units below a directory (implicit stack):
cd live/nonprod
terragrunt run --all plan
terragrunt run --all apply
```
