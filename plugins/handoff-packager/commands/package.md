# /package

Generates the **handoff bundle** that travels alongside the Product Spine. The
engineer reads this at `/validate` to make a Keep/Refactor/Redesign/Reject decision
in minutes, not hours.

This command is normally invoked by `/package-handoff` (prototype-lane plugin). You
can also call it directly to preview what the engineer will see before formally
handing off.

## Surface and connector requirements

Works on **Claude.ai (Chat), Claude Cowork, and Claude Code**. The GitHub
connector is **required** in Chat/Cowork (no local checkout); in Claude Code,
it's used for the dependency-manifest read when the PR is the only place the
diff lives. Refuse cleanly in Chat/Cowork if the connector is missing.

Connector capabilities used:

- **Repo contents** — read manifest diffs (`package.json`, `pyproject.toml`,
  etc.), read the full branch diff to detect novel patterns.
- **Repo contents** (write) — commit the bundle files under
  `proposals/<branch-slug>/handoff/` to the current branch.

## Workflow

### 1. Confirm preconditions

- Current branch is `prototype/*` (or `$ARGUMENTS` is). If not, refuse.
- A Product Spine exists at `proposals/<branch-slug>/product-spine.md`. If not,
  run `/spine-refresh` first.
- The Spine has no `<placeholder>` markers left in non-optional sections.

### 2. Generate the dependency delta

Compute the diff in dependency manifests since `main`:

- `package.json` / `package-lock.json` / `pnpm-lock.yaml` / `yarn.lock` →
  added/removed/upgraded packages.
- `pyproject.toml` / `requirements.txt` / `poetry.lock` → same.
- `Cargo.toml` / `Cargo.lock` → same.
- `go.mod` / `go.sum` → same.

For each added package, fetch (or infer from the lock file) the license and
note if it's on the team's approved-licenses list. Surface anything that isn't.

Write to `proposals/<branch-slug>/handoff/dependency-delta.md` with a table:

| Package    | Version | Change   | License | Approved? | Purpose (inferred from import sites)       |
| ---------- | ------- | -------- | ------- | --------- | ------------------------------------------ |
| `foo`      | `1.2.3` | added    | MIT     | yes       | `apps/<product>/backend/services/order.ts` |

### 3. Generate the novel-patterns report

For the diff against `main`, flag code that doesn't match existing patterns:

- New top-level directories
- New import sources (a library imported nowhere else in the repo)
- New architectural primitives (a new "service", new "context", new "store"
  pattern when the repo has an existing one)
- New testing frameworks or runners
- New build/lint/format tooling
- Custom abstractions where a standard one exists in the codebase

Write to `proposals/<branch-slug>/handoff/novel-patterns.md`. For each finding:
- **What's novel:** one sentence
- **Where it appears:** file paths
- **What the existing pattern is:** evidence from existing code (with paths)
- **Why this might be intentional:** Claude's best guess

The engineer uses this to decide whether each novelty is intentional (Keep) or
should be reworked (Refactor/Redesign).

### 4. Generate the plugin-violations report

Run each house-rule plugin in **production-mode** (strict) against the prototype
branch and collect what would fail:

- `house-style` lint findings
- `security-rails` warnings (non-blocking ones that would block in production)
- `spec-driven-dev` Spine-completeness issues
- `always-test` test-coverage gaps

Write to `proposals/<branch-slug>/handoff/plugin-violations.md`. Group by plugin,
list each violation with file path and line.

This is the **single best signal** for Keep vs Refactor. A clean report = Keep.
Hundreds of findings = Refactor minimum.

### 5. Confirm Spine completeness

Re-verify the Spine has:

- Intent filled in (not a placeholder)
- UX section addressed (or "n/a — backend-only" explicit)
- Surface section addressed (or "n/a — UI-only" explicit)
- Architecture: components, dependencies, assumptions, lane-aware notes
- At least one Open Question OR an explicit "none — all decisions resolved"

If any are missing, fail this command and tell the caller to run `/spine-refresh`
or to fix the Spine by hand before retrying.

### 6. Write the bundle index

Write `proposals/<branch-slug>/handoff/README.md`:

```markdown
# Handoff bundle — <branch slug>

Generated by `handoff-packager` on <date>.

- **Product Spine:** [../product-spine.md](../product-spine.md)
- **Preview URL:** <link>
- **Dependency delta:** [dependency-delta.md](dependency-delta.md) — N packages added, M license concerns
- **Novel patterns:** [novel-patterns.md](novel-patterns.md) — N findings
- **Plugin violations:** [plugin-violations.md](plugin-violations.md) — N findings (strict mode)

Read the Spine first. Read this bundle second. Make a decision.
Do not read the whole branch.
```

### 7. Report back

Single chat message with:

- Bundle path
- Counts for each report (dependencies added, novel patterns, plugin violations)
- One-sentence Keep/Refactor/Redesign/Reject hint based on the counts (e.g.
  "Plugin violations are low and no novel patterns — likely Keep candidate" or
  "47 plugin violations and 3 new architectural primitives — Refactor or Redesign
  likely").

## Things to avoid

- **Do not modify code** or the Spine. This command only writes under
  `proposals/<branch-slug>/handoff/`.
- **Do not pre-decide** the validation outcome. The hint at the end is a hint,
  not a verdict.
- **Do not skip reports** because they're empty. An empty report is information —
  write the file anyway with "no findings".
- **Do not include credentials, env-var values, or any secret** in any report,
  even if you saw them in the code.
